## Architecture invariants

These are load-bearing. Violating them silently corrupts state under load.

**Worker jobs never mutate live world state.** They receive `const World&` and a private `CommandBuffer`. All mutation flows through `EngineImpl::commitBuffer()` (or the sharded variant; see below) which runs after a batch's latch has fired. If you add a new write path into `EntityStorage` outside of the commit phase, you've broken determinism and introduced a data race. The one explicit exception is `EntityStorage::reserveHandle()`, guarded by its own internal mutex; it only manipulates slot allocation, never the archetype chunks themselves.

**Commit order is submission order, not execution order.** `JobSystem` workers race freely; the per-system `std::vector<CommandBuffer>` in `SystemContextImpl` is authoritative. `parallelFor` resizes that vector up front so pointers into it stay valid while jobs run â€” don't switch it to `emplace_back` inside the submit loop.

**The public surface (`include/threadmaxx/`) is the contract.** Everything in `src/` is PImpl'd behind `Engine`/`World` and can change freely. Resist adding new public headers; prefer extending existing ones over exposing internals. `EngineImpl` / `WorldImpl` are reachable from `World::impl_()` for engine-internal use only.

**Systems declare reads/writes; the engine groups them into waves.** `ISystem::reads()` and `ISystem::writes()` return `ComponentSet`s. Defaults are `ComponentSet::all()`, which forces strict registration-order sequential execution. Overriding them lets non-conflicting systems share a wave. The conflict rule is `Wâˆ©W âˆ¨ Wâˆ©R âˆ¨ Râˆ©W`. Waves are recomputed in `registerSystem`. Same-wave systems write into their own `SystemContext` buffers; commits happen in registration order after the wave finishes.

**`ComponentSet` plays a dual role.** Bit positions describe both *scheduling categories* (`reads()`/`writes()`) and *per-entity presence* (`componentMasks()` dense span). `EntityStructural` appears only in scheduling sets â€” it represents "this system touches the set of live entities" (i.e. spawn/destroy).

**Chunked archetype storage.** Entities live in `ArchetypeChunk` instances keyed by their per-entity `ComponentSet`. One chunk per unique mask; chunks own the dense parallel arrays. Bit changes trigger a physical migration during commit (swap-and-pop out of source, push into destination, update slot record). The slot holds `(generation, alive, reserved, archetypeIndex, row)`. Source: `include/threadmaxx/internal/Archetype.hpp` + `src/Archetype.cpp`; mask-change handlers funnel through `EntityStorage::setMaskAndMigrate(handle, newMask)`.

**Legacy parallel-vector view (`World::transforms()`, etc.) is a lazy stitched cache.** First read after any mutation walks chunks in creation order and concatenates per-component values; subsequent reads are cache hits. Mutations flip `stitchedDirty_` (atomic). Cache rebuild is mutex-protected (double-checked locking) â€” reads from non-sim threads are safe with `singleThreadedCommit = true` between waves, unsafe during a sharded Pass C. Iteration order is **archetype-creation order Ã— spawn-order within archetype**, not global spawn order â€” code that needs entity-ordinal stability across mask changes must look up by `EntityHandle`.

**Chunk-iteration query helpers.** `forEachChunk<Required...>(ctx, fn)` in `Query.hpp` is the chunk-iteration foundation; `forEachWith<C...>` and `forEachWithCached<C...>` are the per-entity sugar built on top. All three walk chunks via `ctx.worldView()` (the wave-scoped read-only view) and dispatch via `detail::getChunkSpan<C>`. When adding new query helpers or extending existing ones to a new component: walk chunks, not the stitched view. Mask checks happen once per chunk, not once per entity.

## Adding a new built-in component

This is the one operation that crosses every layer. Missing any one of these compiles cleanly but corrupts state at runtime.

1. Define the POD in `include/threadmaxx/Components.hpp`.
2. Add a `Component::Foo` enum value (bit position past the highest currently allocated; underlying type is `std::uint64_t`) and extend `ComponentSet::all()`'s mask.
3. Add a `std::vector<Foo>` to `ArchetypeChunk` (`Archetype.hpp`). Extend the per-mask `if (c.mask.has(Component::Foo))` branches in `Archetype.cpp` (`insert`, `removeSwapPop`, `migrate`) and the corresponding `mutFoo` on `EntityStorage`. `EntityStorage::spawn` / `materializeReserved` take all dense-component values by const-ref before `ComponentSet initialMask` â€” thread the new one through.
4. Extend `CmdSpawn` (add a `Foo foo` field), add `CmdSetFoo`, all four `spawn` overloads, the matching `CommandBuffer.cpp` method, and `defaultSpawnMask` if presence auto-derives. Add the type to `bundle()`'s switches and a `Foo foo` member on `Bundle`. Extend `addComponent<T>` / `removeComponent<T>` `if constexpr` chains in `CommandBuffer.hpp`.
5. Handle the new variant alternatives in `EngineImpl::commitBuffer` (`std::visit`). Setters that attach a presence bit on write must call `storage.setMaskAndMigrate(handle, newMask)` *before* writing the value via `mutFoo`. For tag-only categories use `addTag` / `removeTag`.
6. Add the read accessor + lazy stitched span on `World` (`tryGetFoo` lowers to `lookupChunkValue<Foo>`; `foos()` returns the stitched view from `EntityStorage::Foos()`). Add the `if constexpr` arm in `detail::worldComponentBit` / `worldTryGetSpanElement` so `World::has<Foo>` / `get<Foo>` work.
7. If rendering-relevant, populate `RenderInstance` in `EngineImpl::buildRenderFrame` with `if (c.mask.has(Component::Foo))` gating.
8. Extend `Query.hpp::detail::getSpan` (stitched-view path), `componentBit` (mask lookup), and `getChunkSpan` (chunk path).
9. Add `serialize` / `deserialize` overloads in `Serialization.hpp`, append a `std::vector<Foo>` field to `WorldSnapshot`, extend its serialize/deserialize bodies, AND extend the `ComponentSet` deserialize fallthrough. Bump `kWorldSnapshotVersion`.
10. Mirror the new dense array in `World::snapshot()` (`src/World.cpp`).
11. Add a Doxygen brief to the POD and any new public methods.
12. Extend `EntityStorage::ensureStitched()` with the new stitched-view vector + `fill()` call.
13. **(B30)** The new `mutFoo()` setter MUST set `c.hashDirty = true` on the touched chunk before returning. Extend `hashChunkContent()` in `src/EngineImpl.cpp` with an `if (c.mask.has(...)) mixVecBytes(c.foos.data(), sizeof(Foo))` arm in component-enum order.

The `ArchetypeChunk` `insert` / `removeSwapPop` / `migrate` triple is the bit most often forgotten â€” every component vector needs a parallel write across all three.

**Tag-only components** (no dense data; presence-bit only) skip steps 3, 6 (no `tryGetFoo`), 8, 9 (no dense vec), 10, 12, 13. Just add the enum bit + extend `ComponentSet::all()` + the deserialize fallthrough. Game code drives via `cb.addTag` / `cb.removeTag` / `world.hasTag`.

**`UserComponent<T>`** (`include/threadmaxx/UserComponent.hpp`) skips ALL thirteen steps â€” game-side types the engine never names. Register via `Engine::registerUserComponent<T>()` (assigns bits from 16 upward; idempotent on `typeid(T)`). Storage: `ArchetypeChunk::userColumns` (a `std::vector<UserComponentColumn>` of `{ bit, stride, std::vector<std::byte> bytes }`). Public API is free-function-shaped: `addUserComponent<T>(cb, id, e, v)`, `removeUserComponent`, `user::has`, `user::tryGet<T>`, `user::chunkSpan<T>`. `World::locate(handle) -> ArchetypeLocation` lets you jump straight to `(archetype, row)`. Out-of-scope: serialization, cross-process bit stability, non-trivially-copyable types â€” all game-side responsibility. Consider this path before extending the built-in recipe for an engine-internal feature.

## Render frame lifetime

`EngineImpl::buildRenderFrame()` writes into the back of a double-buffered pair and publishes via `frontIndex_.store(back, release)`. The `RenderFrame::instances` span points into engine-owned vectors â€” renderers must finish using it before `submitFrame` returns or copy what they need. Today `submitFrame` is called synchronously from the sim thread immediately after the swap; the atomic swap is the synchronization point if a future renderer reads from another thread.

`RenderFrame::alpha` is the wall-clock fraction (0..1) past the last committed tick. `step()` always submits with `alpha=0`. `run()` additionally calls `submitInterpolatedFrame(alpha)` which mutates `alpha` on the current front frame and re-submits â€” no rebuild, since world state is unchanged between ticks.

`RenderFrame::prevTransforms` is a span parallel to `instances`, carrying each render-eligible entity's previous-tick transform (or current on first visible frame). Gives a clean `lerp(prev, current, alpha)` with no spawn special-case. Backed by `prevTransformByIndex_` + `prevTransformGenByIndex_` (flat vectors keyed by entity index; generation guard filters stale entries).

## Render contract (renderer-neutral PODs)

`include/threadmaxx/render/` â€” every type is a renderer-neutral POD; the engine never positions Camera/Light/etc. as per-entity dense storage. Camera/Light/MeshSkinned/AnimationPose are deliberately NOT per-entity components (cameras/lights are few, AnimationPose is ringbuffered, `RenderFrameBuilder` fits them better than `EntityStorage`; rationale in `doc/render_contract.md`).

- `Camera.hpp` â€” POD with `ProjectionMode`, view + projection matrices (column-major). **Projection MUST be Vulkan-style (NDC z âˆˆ [0, 1]): `m22 = f/(n-f)`, `m32 = fn/(n-f)`.** GL-style (`m22 = (f+n)/(n-f)`) clips near-camera vertices in Vulkan and produces visible artifacts on close geometry. `RenderFrame::cameraIndexById(id) -> optional<uint8_t>` maps back to the index in `RenderFrame::cameras` (= bit in `cameraMask`); cap `kMaxCameras = 32`.
- `Light.hpp` â€” `LightType` (Directional/Point/Spot), color/intensity, range, cones, `castsShadow`.
- `DrawItem.hpp` â€” `DrawItem` + `MeshSkinnedRef` / `AnimationPoseRef` / `MaterialOverride`. `cameraMask` is a 32-bit bitset.
- `DebugGeometry.hpp` â€” `DebugLine`, `DebugPoint`, `DebugText`. `DebugText::text` is a borrowed `string_view`; producer owns lifetime to the next frame swap. `RenderFrameBuilder::addDebugText(pos, string_view, color)` has an owning-string overload that copies into a per-builder arena.
- `RenderPasses.hpp` â€” `RenderPass { Opaque, Transparent, ShadowCasters, Overlay }`, `kRenderPassCount = 4`, `passIndex(p)`.
- `RenderFrameBuilder.hpp` â€” per-system slice. Move-only; engine owns one per system, reset before each tick, merged after every postStep + event drain.
- `Visibility.hpp` â€” `extractFrustum(camera) -> Frustum`, `intersectsFrustum` AABB p-vertex test, `cullByFrustum(items, bounds, cameras)` writes `cameraMask` in place (33rd camera silently dropped).
- `InstanceBufferLayout.hpp` â€” `alignas(16) InstanceLayoutEntry` (128 B, std140-friendly). `packInstance(item)` / `packInstances(items, dst)`.
- `UploadRing.hpp` â€” header-only frame-to-frame bump allocator. `setGrowOnOverflow(true)` allows oversize allocations.

## ISystem::buildRenderFrame hook

`ISystem::buildRenderFrame(RenderFrameBuilder&)` is the render-prep hook. Default: empty. Invoked after every `postStep` and event drain, on the sim thread, single-threaded, in registration order. Each registered system gets its own builder (engine-owned, persisted across ticks for zero-alloc steady state); the engine merges every builder into the back frame.

`RenderFrame` exposes both paths simultaneously: the flat `instances` lane (auto-populated from `RenderTag + !DisabledTag` entities) AND the hierarchical fields (populated via the hook). The two coexist with no overlap. Headless renderers consume just `instances`; richer renderers consume the hierarchical fields.

`SystemStats::buildRenderFrameSeconds` is the per-system timing for the hook (reset to 0 each tick, including across paused steps).

## Bundles, spawning, World query sugar

- `bundle(Cs...)` (`CommandBuffer.hpp`) is a variadic factory returning a `Bundle` with the supplied values and `initialMask` = union of the bits. Feed to `CommandBuffer::spawnBundle(...)`.
- `Bundle::with<T>(value)` is the builder method that sets the field AND the matching presence bit in one call. Composable: `Bundle{}.with(Transform{...}).with(Health{...})`.
- Default-mask `spawn(...)` auto-derives `RenderTag` (from `meshId >= 0`) and `Parent` (from `parent.valid()`). The Â§3.1 batch-5 setters (`setHealth` / `setFaction` / `setAnimationStateRef` / `setPhysicsBodyRef` / `setNavAgentRef` / `setBoundingVolume`) attach their presence bit on commit; detach via `cb.removeTag(e, Component::Foo)` or `cb.setComponentMask`. Default `spawn` does NOT attach these bits â€” opt in via `Bundle` or the explicit-mask overload.
- `world.has<T>(handle)` / `world.get<T>(handle)` â€” header-only sugar for the built-in types. `get` asserts presence. Tag-only categories (`StaticTag`, `DisabledTag`, `DestroyedTag`) reject `has<T>` at compile time â€” use `world.hasTag(handle, Component::Tag)` instead.

## ComponentSet width & tag-only components

`ComponentSet` is `std::uint64_t`; 16 bits currently allocated (Transform=0 â€¦ DestroyedTag=15). 48 spare bits before another widening. Serialization writes the full 64-bit field; v1 32-bit snapshots are not loadable.

`StaticTag` / `DisabledTag` / `DestroyedTag` have no dense storage and no commands. `cb.addTag` / `cb.removeTag` race-free single-bit flips that compose across workers. `buildRenderFrame` skips `DisabledTag`; the engine never auto-destroys on `DestroyedTag` â€” it's a gameplay-only marker. Tag bits round-trip through `WorldSnapshot`.

## Engine subsystems

### ResourceRegistry

`include/threadmaxx/Resource.hpp` â€” engine-owned, thread-safe under a single internal mutex. Values are type-erased `std::shared_ptr<void>` keyed by `std::type_index`. Sized for setup-time registration and per-frame lookups, not high-throughput concurrent inserts â€” async loaders should do I/O off-thread and call `add()` when ready. Stale handles never alias new ones (slot generation bumps on every `remove`).

Opt-in refcount path: `addRefCounted<T>(value)` returns a `ResourceHandle<T>` (refcount=1); `acquire(id)` bumps; `~ResourceHandle` decrements; on zero refcount the slot is freed AND its generation bumps. Legacy-added slots are marked `refCounted=false` and `acquire` refuses them. `refCount(id)` reports 0 for non-refcounted or stale slots. `ResourceHandle<T>::get()` / `operator->` / `operator*` are the indirection sugar (`get` returns `const T*` or nullptr).

### Resource loaders + hot reload + preload

`IResourceLoader` (`Resource.hpp`) is the per-tick pump for asset I/O. Registered via `Engine::addResourceLoader`. The engine calls `loader->update(engine)` once per `step()` after the last `postStep` commits and before the reservation reap / event drain. Teardown in reverse-registration order during `shutdown()`. Engine never spawns threads for loaders â€” each owns its own pool.

Optional virtuals: `onShutdown(Engine&)` (reverse-registration teardown), `markStale(index, generation, type)` (dispatched by `Engine::markResourceStale<T>(id)`; loader publishes `AssetReloaded{old/new index+gen, type}` on `events<AssetReloaded>()` after installing the replacement â€” use `AssetReloaded::matches<T>(id)` to filter), `cancel(Engine&)` (called before `update()` so a loader can drop newly-stale requests in the same tick; `LoaderStats::cancelled` aggregates), `stats() -> LoaderStats` (pending/inFlight/ready/failed + memory; `Engine::aggregateLoaderStats()` sums them).

`Engine::preloadUntil(predicate, timeout = 5s)` pumps every loader's `update()` in a yield loop until predicate or timeout. Simulation doesn't advance. Sim-thread only.

### HierarchySystem

`makeHierarchySystem(HierarchyConfig{})` propagates world transforms for `Parent`-attached entities. Runs single-threaded in `ctx.single()`; DFS-with-memoization plus a cycle-safety `onStack` guard. Scratch state is system-member, reused across ticks. Composition: `position = parent.position + rotate(parent.orientation, local.position)`; `orientation = parent.orientation * local.orientation`; `scale = local.scale` by default (`HierarchyConfig::propagateScale = true` opts in to component-wise chaining). See `doc/hierarchy.md`.

Register *after* any user system that writes `Transform`. Both declare `writes = {Transform}` â†’ different waves; hierarchy sees the writer's commits.

### JobSystem

`src/JobSystem.{hpp,cpp}` â€” per-worker work-stealing deques. Submit round-robin via an atomic counter; idle workers `try_lock`-steal LIFO from siblings' back. `waitIdle` synchronizes on a separate mutex/CV. `outstanding_` is `acq_rel` for cross-thread visibility; `notify_all` on `doneCv` when it hits zero.

Per-worker `ownPops` / `stolenJobs` / `histogram` are `std::atomic<uint64_t>` with relaxed ordering â€” when adding a new per-worker counter follow that pattern. Per-worker counters are touched by the owning thread (own pop/histogram) or by the stealing thread (writing its OWN slot for `stolenJobs`); reads from `stats()` happen from any thread.

`JobSystemStats::jobDurationHistogram` is 16 log2-Âµs bins; `JobSystem::binFor(ns)` is static for testability. `JobSystem::outstanding()` reports live in-flight for queue-depth sampling.

Determinism is preserved because commits are submission-ordered (worker execution order doesn't matter). Stress test: `tests/job_system_stress_test.cpp`.

**`JobLatch` is the TSAN-clean replacement for `std::latch`.** Lives in the anonymous namespace at the top of `src/EngineImpl.cpp`. libstdc++'s `std::latch::wait()` lowers to a futex that ThreadSanitizer cannot see through. `JobLatch` uses `std::ptrdiff_t` under a mutex+CV â€” every `count_down` takes the lock. **Don't re-introduce `std::latch` here without a TSAN re-run; don't re-introduce the lock-only-on-last-decrement form without 100Ã— stress validation under TSAN** (the "optimized" variant had a missed-wakeup hazard).

`JobPriority { High, Normal, Low }` (`System.hpp`). `parallelFor` overloads accept it; no-priority overloads default to `Normal`. Workers scan deques in priority order. Backwards-compatible â€” collapses to single-deque behavior when every job is `Normal`.

### Lifecycle hooks

`ISystem` exposes three: `preStep` (serial, registration order, before any wave), `update` (parallel within a wave), `postStep` (serial, registration order, after every wave). `preStep`/`postStep` default to no-op; commits flush immediately on the sim thread.

### ScratchArena

`include/threadmaxx/ScratchArena.hpp` is a chained-slab bump allocator. Pass via the three-arg `parallelFor` / `single` overloads alongside the `CommandBuffer`. Slabs survive across waves; reset rewinds the bump pointer. `allocate<T>` static_asserts that `T` is trivially destructible. Wave-scoped lifetime, not engine-scoped.

### EventChannel

`include/threadmaxx/EventChannel.hpp` â€” typed double-buffered queue. `Engine::events<T>()` is the factory; channels are owned in `EngineImpl::eventChannels_` keyed by `std::type_index`. The map is mutex-guarded. **Warm cross-thread channels at setup** by calling `events<T>()` on the sim thread before workers / watchdogs touch them.

`emit` is lock-free (atomic Treiber stack with CAS prepend; `exchange(nullptr)` + reverse on drain restores per-thread FIFO). Subscriber list is mutex-guarded but only touched on subscribe / unsubscribe. `pendingCount()` is approximate under concurrent emit.

Subscriptions: `subscribe(fn) -> SubscriptionId` / `unsubscribe(id)` for persistent; `subscribeScoped(fn) -> Subscription` for RAII (move-only, type-erased, auto-detaches; holds a `weak_ptr` so it safely no-ops if the channel dies first; `reset()` / `valid()`). Tick-end `drain()` invokes each callback once per back-buffered event BEFORE the front/back swap. Subscriber list is snapshotted before invocation â€” a callback that subscribes/unsubscribes doesn't invalidate iteration.

### Pause & time scale

`Engine::setTimeScale(s)` clamps negative `s` to zero; `dt` becomes `fixedStepSeconds * s`. `tick()` and `simulationTime()` advance independently of time scale. `setPaused(true)` makes `step()` a no-op (zeros per-tick stats); `run()` keeps re-submitting the current front frame so the renderer doesn't freeze.

### Reserved handles

`Engine::reserveEntityHandle()` / `SystemContext::reserveHandle()` allocate via `EntityStorage::reserveHandle()` under `reservationMtx_`. `EntityHandle::valid()` is true immediately but `world.alive(handle)` is false until `materializeReserved` runs during commit (triggered by `cb.spawn(handle, ...)`). Unmatched reservations are reaped in `discardAllReservations` at step end (bumps generation, returns slot to free list).

Batch form: `Engine::reserveEntityHandles(count, span)` / `SystemContext::reserveHandles(count, span)` / `EntityStorage::reserveHandles` acquire the mutex once. Returns the number actually written (`min(count, span.size())`).

### SpatialHash

`SpatialHash<Payload>` (header-only, `SpatialHash.hpp`) â€” uniform-grid index keyed by 3D cell coordinate. Not thread-safe: insert from a single thread (usually `preStep`); queries safe from workers once built. `clear()` keeps bucket allocation so steady-state per-tick rebuilds pay zero allocations after the first tick. Engine doesn't own one â€” user composes it.

### Serialization

`include/threadmaxx/Serialization.hpp` is header-only. Per-component `serialize` / `deserialize` for every built-in plus `Vec3`, `Quat`, `ComponentSet`, and the `WorldSnapshot` POD. `World::snapshot()` (`World.cpp`) copies the engine's dense arrays into a `WorldSnapshot`. Binary format: `[magic 'SXMT'][version u32][count u64][N dense arrays each prefixed by u64 length]`; deserialize rejects mismatched magic/version. `kWorldSnapshotVersion = 2`. Restoration is game-side via `cb.spawn(...)` â€” the engine never bypasses commit.

### Logger

`ILogger` (`Logger.hpp`) is a single `log(level, message)` virtual. The engine routes startup/shutdown/registration/loader-error messages through whichever sink is installed via `Engine::setLogger`; `nullptr` restores `DefaultLogger` (writes to `std::cerr` at `Warn+`). `EngineImpl::logger()` returns the active sink. The engine doesn't take ownership.

### registerSystemAt

`Engine::registerSystemAt(position, system)` inserts at a specific index in registration order (clamped to `[0, registeredSystemCount()]`). Calls `onRegister(world)` and `rebuildWaves()`. Returns the actual insertion index. For tests / mod loaders that need to inject a system at a known point.

### IRenderer::onResize

`IRenderer::onResize(w, h)` is a default-no-op virtual; `Engine::notifyResize(w, h)` forwards window resize events into the currently installed renderer. No engine state changes. Sim-thread by convention.

`Engine::workerCount()` is the public accessor for the worker pool size â€” cheaper than going through `jobSystemStats().workerCount` (no atomic loads / no histogram merge). Use for sizing per-worker scratch and ring-buffer slots.

## Task graph (named tag edges) & wave scheduler

Three optional virtuals on `ISystem`:
- `dependencies() -> span<const TaskTag>` â€” tags this system *consumes*.
- `provides() -> span<const TaskTag>` â€” tags this system *produces*.
- `preferredGrain() -> uint32_t` â€” hint to `pickGrain` when `parallelFor` passes `grain=0`. Pinned per-system.

`TaskTag { string_view name; uint64_t hash; }` (`TaskTag.hpp`) â€” constexpr FNV-1a hash; `operator==` compares `name`. Hash collisions degrade to extra string comparisons, never to wrong scheduling.

`EngineImpl::rebuildWaves` does topological-sort-then-pack:
1. Edges: rw-conflict `a < b` â†’ `a â†’ b`; tag overlap `a.provides âˆ© b.dependencies` â†’ `a â†’ b` (both directions checked).
2. Kahn picks zero-in-degree systems in registration order. On stall (tag cycle), the lowest-index stuck system has its tag-only incoming edges dropped, logged via `ILogger`, Kahn resumes.
3. Greedy wave-pack: earliest wave where predecessors are done AND no rw-conflict with anyone in the wave.

Bit-for-bit unchanged when no tags are declared. With tags, consumers may land later than first-fit would have given; producers are never delayed by consumers.

`Engine::taskGraphSnapshot() -> std::vector<TaskGraphNode>` exposes the graph for HUD diagnostics / Graphviz / test assertions.

## Adaptive tuning (ADAPTIVE_TUNING.md)

`include/threadmaxx/Tuning.hpp` is the public surface for the Phase-T tuner. Three types: `SystemGrainOverride { string systemName; uint32_t preferredGrain; }`, `TuningPatch { vector<SystemGrainOverride> grainOverrides; }`, and `ITuningPolicy { void observe(EngineStats, span<SystemStats>, JobSystemStats); optional<TuningPatch> propose(); }`. `Engine::setTuningPolicy(p)` / `tuningPolicy()` install / inspect; `setTuningPolicy(nullptr)` detaches and discards any staged patch.

**Call sites in `step()`** â€” `observe()` and `propose()` fire once per step AFTER the trace-sink callback (same `EngineStats` / `SystemStats` / `JobSystemStats` view a sink would see). Any returned non-empty patch is stored in `EngineImpl::pendingPatch_`. `applyPendingTuningPatch()` runs at the top of the NEXT `step()` BEFORE preStep â€” never mid-wave. Patch application overwrites `EngineImpl::systemPreferredGrain_[i]` for each matched name (linear name scan); unknown names log `[tuning] grain override for unknown system 'X' ignored` at Warn. Applied changes log `[tuning] 'X' preferredGrain old -> new` at Info.

**Determinism contract** â€” same input + same scripted patch stream = bit-identical `commitHash` stream (tuning is a scheduling knob; never touches storage or commit order). T6 will add `TuningTrace` + `TuningMode::Scripted` for networked / replay. The mechanism cost when no policy is installed is one null-pointer check per step.

**`AdaptiveGrainPolicy`** (`include/threadmaxx/AdaptiveGrainPolicy.hpp`, header-only). Built-in `ITuningPolicy` that adjusts per-system `preferredGrain` toward a hold band `[targetSubJobMicros/2, targetSubJobMicros*4]`. Per-system EWMA of `SystemStats::avgSubJobMicros` drives the direction; hysteresis = `minSamplesPerChange` same-direction observations before a fire + `cooldownTicks` between consecutive fires on the same system. Îµ-greedy random one-step walks (probability `explorationEpsilon`) escape stuck local minima. RNG is seeded deterministically (`randomSeed ^ currentTick`) so two runs against identical input produce identical patch streams. Config defaults (`cooldownTicks=60`, `targetSubJobMicros=200`, `stepSize=1.5`, `initialGrain=64`) are tuned conservatively â€” slow to act, no oscillation under noisy workloads. Bench gate: `bench/adaptive_tuning_bench --mode=tiny-fanout` lands within +10% of the offline grain-sweep optimum (last empirical: +0.14%).

## Cancellation, budgets, priorities

**Tick budget + skip policy.** `Engine::setTickBudget(seconds)` caps the wall-clock per-tick spend. The engine samples after every wave; if elapsed > budget AND `setSkipPolicy(SkipPolicy::Budget)` (the default), `overBudget_` flips and subsequent waves' `ISystem::skippable()` systems are skipped. `preStep` / `postStep` / `buildRenderFrame` are NEVER skipped.

`SkipPolicy::Scripted` ignores `tickBudget_` and consults `scriptedSkips_` (populated via `Engine::pushScriptedSkip(tick, name)`). Networked / lockstep games run `Budget` on the authoritative server, drain `EventChannel<SystemSkipped>`, broadcast the log, clients replay with `Scripted` â€” world hashes match.

`SystemContext::shouldYield()` exposes the over-budget flag â€” a single atomic load, cheap to poll inside long `parallelFor` bodies.

`SystemSkipped { tick, systemName, reason }` event (`SkipPolicy.hpp`) emitted on skip; reason is `"budget"` or `"scripted"`.

## Commit path + hash determinism (v1.3)

`Config::singleThreadedCommit = true` (default) â€” wave commits go through `commitBuffer`. Pre/postStep hooks always do regardless.

`Config::singleThreadedCommit = false` opts into `commitBuffersSharded` (three passes: build migrating-entity set; sim-thread classify + apply migrations + bin value-only commands per chunk; parallel-apply bins). **Sharded never beats single-threaded on tested workloads** (serial classifier ~130 ns/cmd exceeds 4-way apply's 50 ns/cmd recovery) â€” keep the default. Documented fallback under profiler-confirmed contention. Auto-falls-through to single-threaded when `totalCommands < 256` OR `totalValueOnly == 0` OR `archetypeChunkCount() < 2`.

**Per-archetype hash rollup is the v1.3 commit-hash contract** (`Config::legacyCommitHash = false`, the default). Every `ArchetypeChunk` carries `mutable uint64_t cachedHash + mutable bool hashDirty`; `EngineImpl::finalizeCommitHash()` re-hashes dirty chunks at end of step (parallel when â‰¥ 2 dirty), folds sorted-by-`mask.bits()` into `EngineStats::commitHash`. v1.2 â†’ v1.3 contract: "same command stream" â†’ "same final per-archetype state." External clients must re-record reference hashes; `Config::legacyCommitHash = true` keeps the v1.x byte-mix during deprecation (slated for removal in v1.4).

**Chunk dirty-marking is load-bearing.** Every mutation path MUST set `chunk.hashDirty = true`. Covered: `ArchetypeTable::insert` / `removeSwapPop` / `migrate` (the latter via the other two), every `EntityStorage::mut*()` setter. When adding a new chunk-mutating method to `ArchetypeTable`, set dirty at the end. `hashDirty` is a plain `bool` â€” safe because writers are sim-thread-only or per-chunk-disjoint workers with a `JobLatch` acquire-release before the reader. Concurrent multi-thread chunk writes would require serialization or atomic upgrade.

`hashChunkContent` is FNV-1a-64 over: `mask.bits` â†’ `entities.size` â†’ `entities[0..]` â†’ each built-in dense vec in component-enum order (mask-gated) â†’ each user column in ascending-bit order (`col.bit`, `col.stride`, `col.bytes`). `finalizeCommitHash()` sorts chunks by `mask.bits()` ascending before folding (empty chunks included).

Adding a new command variant requires arms in both `applyCommandImpl` and `hashCommandImpl` at the top of `src/EngineImpl.cpp`. Both commit paths go through these helpers â€” a missing arm fails to compile rather than silently corrupting state. New variants auto-fall-through to the global lane (`commandIsMigrating` returns true for anything not in `CmdSetTransform / CmdSetVelocity / CmdSetAcceleration / CmdSetUserData`).

`Config::logCommitHashEvery = N > 0` logs `commitHash tick=<T> hash=0x<16hex>` via `ILogger@Info` every N ticks â€” pinpoints a divergent tick for local repro. Default `0` is silent / zero-cost.

## Stitched-cache safety

`EntityStorage::ensureStitched()` is mutex-protected (double-checked locking on `stitchedDirty_`). Fast path (cache clean) only pays the atomic acquire on `stitchedDirty_`; slow path (rebuild) is serialized. Required to make `world.transforms()` etc. safe from many concurrent worker reads in the same wave.

## WorldView

`SystemContext::worldView() -> const WorldView&` â€” wave-scoped read-only view of chunk storage. Caches chunk pointers + entity counts at wave start; shared across every system in the wave. Engine rebuilds it between waves (commits may shift chunk inventory). Capture by reference and reuse across multiple `parallelFor` / `single` calls without paying repeated indirection through `World::archetypeChunk(i)`. `EngineImpl` rebuilds `worldView_` inside `step()` before preStep, between preStep commits, at the top of every wave, and between postStep commits.

## MaskCache

`Query.hpp::MaskCache` â€” user-owned vector of dense indices matching a `ComponentSet`. `forEachWithCached<...>` iterates the cache, skips out-of-range entries, no per-entity mask test in the hot loop. Rebuild discipline is on the caller: `cache.rebuild(world, required<T...>())` in `preStep` when the required set may have a new or removed entity. The bounds check protects against destroys between rebuilds but cannot detect a mask flip to no-longer-match. `MaskCache::reserve(n)` / `capacity()` pre-warm the index storage; `clear()` preserves capacity.

## Sub-job dispatch in forEachChunk

`forEachChunk<Required...>` splits oversized chunks into multiple sub-jobs so workers fan out across the rows of a single huge chunk. `kForEachChunkSubJobThreshold = 1024` (public constexpr in `Query.hpp`); chunks at or below stay a single job, above splitting kicks in. `detail::chunkSubJobBudget(rowCount, workers)` picks per-sub-job rows as `max(threshold, ceil(rowCount / (workers * 8)))`. The callback fires once per sub-job; spans across sub-jobs sum to the full chunk's rows; every row visited exactly once. `SystemContext::workerCount() -> uint32_t` is the cached pass-through to `Engine::workerCount()` used to compute the budget.

## Async snapshot / trace sink

`FileTraceSink::setAsync(bool)` spawns an internal writer thread + queue + mutex/CV. `onFrame` copies the borrowed `FrameSnapshot` into owned storage (`OwnedFrameSnapshot` in `Telemetry.cpp` is the bridge), enqueues, returns. Joined on `setAsync(false)`, `onShutdown`, or destruction.

`Engine::snapshotAsync(callback)` captures `world().snapshot()` synchronously on the sim thread (the snapshot copy must be sync â€” `WorldView` caches chunk pointers, not contents, so a mid-snapshot commit would race), then posts the user's callback onto a dedicated engine-owned background thread (`snapshotWorker_`, lazily spawned, joined in `shutdown` / `~EngineImpl`).

Snapshot consistency: reflects state at the moment `snapshotAsync` was called. Callbacks must not call back into the engine's mutation API. When adding new async I/O sinks, follow the same pattern: deep-copy the borrowed span, enqueue, drain on a dedicated worker, join in dtor.

## Telemetry / FrameSnapshot

`Engine::frameSnapshot()` returns `FrameSnapshot{EngineStats, span<const SystemStats>, JobSystemStats}` â€” consistent across the three. `writeJsonLines(os, snap)` serializes one snapshot as a single `\n`-terminated JSON object with `"commit_hash":"0xâ€¦"` (16-char hex). `ChromeTraceWriter` is the streaming companion: ctor writes `[`, `emit(snap)` appends one `step` record + one per system per call (`args.commit_hash` carries the hash), dtor writes `]`. Systems span is invalidated by `registerSystem` / `registerSystemAt` / `shutdown`.

`SystemStats` extras: `waitSeconds` (latch wait time across `parallelFor`), `peakQueueDepth` (max `JobSystem::outstanding()` right after submit), `buildRenderFrameSeconds`, plus the ADAPTIVE_TUNING.md T3 pair `avgSubJobMicros` (1/16-EWMA of per-sub-job lambda Âµs; persists across steps, 0 if the system never called `parallelFor`) and `subJobsLastStep` (parallelFor sub-jobs dispatched this step). Both fields are emitted in `writeJsonLines` (`avg_sub_job_us`, `sub_jobs`) and in `ChromeTraceWriter` per-system args, and surfaced in `perf_audit_rpg_demo`'s top-systems table.

`include/threadmaxx/Telemetry.hpp` opt-in extras:
- `ITraceSink::onFrame(FrameSnapshot)` â€” installed via `Engine::setTraceSink(ITraceSink*)`. Engine never takes ownership; sink must outlive the engine.
- `FileTraceSink` â€” rolling Chrome-trace JSON. Rotates when file exceeds `Config::rotationBytes` (default 64 MiB); `pathTemplate` may contain `%N` or be a stem. Silent no-op on open failure.
- `HudTraceSink` â€” seqlock-protected latest-snapshot sink. `tryGet(out)` is lock-free and torn-write-free. **The seqlock has an inherent TSAN-visible race on `data_` BY DESIGN; suppressed in `cmake/tsan.supp`. Don't "fix" it with mutexes** â€” that defeats the purpose. Activate with `TSAN_OPTIONS=suppressions=<path>/cmake/tsan.supp`.
- `FrameBudgetWatcher` â€” `ISystem` whose `postStep` emits `BudgetExceeded` when `lastStepSeconds` > target.
- Stall watchdog â€” `Engine::setStallTimeout(seconds)` spawns a background thread that emits `EngineStall` via `events<EngineStall>()`. `stallTimeoutSeconds_` is `std::atomic<double>` (relaxed). Joined on `setStallTimeout(0.0)` or `~EngineImpl`. Zero overhead when disabled.

Add telemetry from a new subsystem: emit a typed event through `events<MyTelemetryEvent>()` and tell sinks to subscribe.

## Performance patterns

- Prefer flat vectors keyed by entity index over hash maps for hot paths.
- Reuse allocations across ticks via `clear()` / `assign()` to preserve capacity.
- Avoid `std::thread` creation in steady-state code â€” dispatch through `jobs_->submit` instead (wave-parallel system updates do this).
- Walk chunks via `ctx.worldView()` and `detail::getChunkSpan<C>` rather than the stitched `world.transforms()` view (the stitched view acquires a mutex on first build; chunk walking is mutex-free and 3â€“5Ã— cheaper at small scales).
- The Â§3.9.3 batch-18 trick: keep `detail::Command` variant alternatives under 48 bytes when possible. `CmdSpawn` and `CmdAddUserComponent` ride `std::unique_ptr<>` wrappers to keep the variant at 64 B (vs 256 B if both were inlined). The `unwrap` helper at the top of `applyCommandImpl` / `hashCommandImpl` dereferences them transparently. If a new variant would push the variant over 64 B, follow the same pattern.
