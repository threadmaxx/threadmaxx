This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository. It captures the current API surface, architecture invariants, and load-bearing conventions. Historical batch logs (every shipped/deferred batch, every audit round, every release-note dump) live in `CLAUDE_ARCHIVE.md` — grep it when you need the deep history.

## Build, run, iterate

```
cmake -S . -B build -DCMAKE_BUILD_TYPE=Release -DCMAKE_EXPORT_COMPILE_COMMANDS=ON
cmake --build build -j
./build/examples/minimal/threadmaxx_minimal          # Ctrl-C to quit
./build/examples/minimal/threadmaxx_minimal 600      # bounded: run exactly 600 ticks then exit
cd build && ctest --output-on-failure                # test suite
```

Useful CMake options:

- `-DTHREADMAXX_WARNINGS_AS_ERRORS=ON` — promotes the GCC/Clang warning set (incl. `-Wsign-conversion`, `-Wconversion`, `-Wshadow`, `-Wold-style-cast`) to errors. The project compiles clean under it; keep it that way.
- `-DTHREADMAXX_BUILD_EXAMPLES=OFF` / `-DTHREADMAXX_BUILD_TESTS=OFF` — narrow the build.
- `-DTHREADMAXX_BUILD_BENCHMARKS=ON` — opt in to `bench/` (not registered with CTest; run binaries manually).
- `-DTHREADMAXX_BUILD_LONG_SOAK=ON` — opt in to `tests/concurrency_soak_long.cpp` (5–6 min runtime).

Always pass `-DCMAKE_EXPORT_COMPILE_COMMANDS=ON` on first configure — without it, clangd can't see include paths or `cxx_std_20` and will report spurious errors.

Tests use a tiny no-dependency harness (`tests/Check.hpp`); one executable per test, non-zero exit means failure. New behavior should land with a test in the same PR — the suite is the contract.

The integration smoke test is `examples/minimal/threadmaxx_minimal`: a successful run prints `[frame]` lines with monotonically increasing ticks and ends with `[ConsoleRenderer] shutdown after N frames`.

## Architecture invariants

These are load-bearing. Violating them silently corrupts state under load.

**Worker jobs never mutate live world state.** They receive `const World&` and a private `CommandBuffer`. All mutation flows through `EngineImpl::commitBuffer()` (or the sharded variant; see below) which runs after a batch's latch has fired. If you add a new write path into `EntityStorage` outside of the commit phase, you've broken determinism and introduced a data race. The one explicit exception is `EntityStorage::reserveHandle()`, guarded by its own internal mutex; it only manipulates slot allocation, never the archetype chunks themselves.

**Commit order is submission order, not execution order.** `JobSystem` workers race freely; the per-system `std::vector<CommandBuffer>` in `SystemContextImpl` is authoritative. `parallelFor` resizes that vector up front so pointers into it stay valid while jobs run — don't switch it to `emplace_back` inside the submit loop.

**The public surface (`include/threadmaxx/`) is the contract.** Everything in `src/` is PImpl'd behind `Engine`/`World` and can change freely. Resist adding new public headers; prefer extending existing ones over exposing internals. `EngineImpl` / `WorldImpl` are reachable from `World::impl_()` for engine-internal use only.

**Systems declare reads/writes; the engine groups them into waves.** `ISystem::reads()` and `ISystem::writes()` return `ComponentSet`s. Defaults are `ComponentSet::all()`, which forces strict registration-order sequential execution. Overriding them lets non-conflicting systems share a wave. The conflict rule is `W∩W ∨ W∩R ∨ R∩W`. Waves are recomputed in `registerSystem`. Same-wave systems write into their own `SystemContext` buffers; commits happen in registration order after the wave finishes.

**`ComponentSet` plays a dual role.** Bit positions describe both *scheduling categories* (`reads()`/`writes()`) and *per-entity presence* (`componentMasks()` dense span). `EntityStructural` appears only in scheduling sets — it represents "this system touches the set of live entities" (i.e. spawn/destroy).

**Chunked archetype storage.** Entities live in `ArchetypeChunk` instances keyed by their per-entity `ComponentSet`. One chunk per unique mask; chunks own the dense parallel arrays. Bit changes trigger a physical migration during commit (swap-and-pop out of source, push into destination, update slot record). The slot holds `(generation, alive, reserved, archetypeIndex, row)`. Source: `include/threadmaxx/internal/Archetype.hpp` + `src/Archetype.cpp`; mask-change handlers funnel through `EntityStorage::setMaskAndMigrate(handle, newMask)`.

**Legacy parallel-vector view (`World::transforms()`, etc.) is a lazy stitched cache.** First read after any mutation walks chunks in creation order and concatenates per-component values; subsequent reads are cache hits. Mutations flip `stitchedDirty_` (atomic). Cache rebuild is mutex-protected (double-checked locking) — reads from non-sim threads are safe with `singleThreadedCommit = true` between waves, unsafe during a sharded Pass C. Iteration order is **archetype-creation order × spawn-order within archetype**, not global spawn order — code that needs entity-ordinal stability across mask changes must look up by `EntityHandle`.

**Chunk-iteration query helpers.** `forEachChunk<Required...>(ctx, fn)` in `Query.hpp` is the chunk-iteration foundation; `forEachWith<C...>` and `forEachWithCached<C...>` are the per-entity sugar built on top. All three walk chunks via `ctx.worldView()` (the wave-scoped read-only view) and dispatch via `detail::getChunkSpan<C>`. When adding new query helpers or extending existing ones to a new component: walk chunks, not the stitched view. Mask checks happen once per chunk, not once per entity.

## Adding a new built-in component

This is the one operation that crosses every layer. Missing any one of these compiles cleanly but corrupts state at runtime.

1. Define the POD in `include/threadmaxx/Components.hpp`.
2. Add a `Component::Foo` enum value (bit position past the highest currently allocated; underlying type is `std::uint64_t`) and extend `ComponentSet::all()`'s mask.
3. Add a `std::vector<Foo>` to `ArchetypeChunk` (`Archetype.hpp`). Extend the per-mask `if (c.mask.has(Component::Foo))` branches in `Archetype.cpp` (`insert`, `removeSwapPop`, `migrate`) and the corresponding `mutFoo` on `EntityStorage`. `EntityStorage::spawn` / `materializeReserved` take all dense-component values by const-ref before `ComponentSet initialMask` — thread the new one through.
4. Extend `CmdSpawn` (add a `Foo foo` field), add `CmdSetFoo`, all four `spawn` overloads, the matching `CommandBuffer.cpp` method, and `defaultSpawnMask` if presence auto-derives. Add the type to `bundle()`'s switches and a `Foo foo` member on `Bundle`. Extend `addComponent<T>` / `removeComponent<T>` `if constexpr` chains in `CommandBuffer.hpp`.
5. Handle the new variant alternatives in `EngineImpl::commitBuffer` (`std::visit`). Setters that attach a presence bit on write must call `storage.setMaskAndMigrate(handle, newMask)` *before* writing the value via `mutFoo`. For tag-only categories use `addTag` / `removeTag`.
6. Add the read accessor + lazy stitched span on `World` (`tryGetFoo` lowers to `lookupChunkValue<Foo>`; `foos()` returns the stitched view from `EntityStorage::Foos()`). Add the `if constexpr` arm in `detail::worldComponentBit` / `worldTryGetSpanElement` so `World::has<Foo>` / `get<Foo>` work.
7. If rendering-relevant, populate `RenderInstance` in `EngineImpl::buildRenderFrame` with `if (c.mask.has(Component::Foo))` gating.
8. Extend `Query.hpp::detail::getSpan` (stitched-view path), `componentBit` (mask lookup), and `getChunkSpan` (chunk path).
9. Add `serialize` / `deserialize` overloads in `Serialization.hpp`, append a `std::vector<Foo>` field to `WorldSnapshot`, extend its serialize/deserialize bodies, AND extend the `ComponentSet` deserialize fallthrough. Bump `kWorldSnapshotVersion`.
10. Mirror the new dense array in `World::snapshot()` (`src/World.cpp`).
11. Add a Doxygen brief to the POD and any new public methods.
12. Extend `EntityStorage::ensureStitched()` with the new stitched-view vector + `fill()` call.
13. **(B30)** The new `mutFoo()` setter MUST set `c.hashDirty = true` on the touched chunk before returning. Extend `hashChunkContent()` in `src/EngineImpl.cpp` with an `if (c.mask.has(...)) mixVecBytes(c.foos.data(), sizeof(Foo))` arm in component-enum order.

The `ArchetypeChunk` `insert` / `removeSwapPop` / `migrate` triple is the bit most often forgotten — every component vector needs a parallel write across all three.

**Tag-only components** (no dense data; presence-bit only) skip steps 3, 6 (no `tryGetFoo`), 8, 9 (no dense vec), 10, 12, 13. Just add the enum bit + extend `ComponentSet::all()` + the deserialize fallthrough. Game code drives via `cb.addTag` / `cb.removeTag` / `world.hasTag`.

**`UserComponent<T>`** (`include/threadmaxx/UserComponent.hpp`) skips ALL thirteen steps — game-side types the engine never names. Register via `Engine::registerUserComponent<T>()` (assigns bits from 16 upward; idempotent on `typeid(T)`). Storage: `ArchetypeChunk::userColumns` (a `std::vector<UserComponentColumn>` of `{ bit, stride, std::vector<std::byte> bytes }`). Public API is free-function-shaped: `addUserComponent<T>(cb, id, e, v)`, `removeUserComponent`, `user::has`, `user::tryGet<T>`, `user::chunkSpan<T>`. `World::locate(handle) -> ArchetypeLocation` lets you jump straight to `(archetype, row)`. Out-of-scope: serialization, cross-process bit stability, non-trivially-copyable types — all game-side responsibility. Consider this path before extending the built-in recipe for an engine-internal feature.

## Render frame lifetime

`EngineImpl::buildRenderFrame()` writes into the back of a double-buffered pair and publishes via `frontIndex_.store(back, release)`. The `RenderFrame::instances` span points into engine-owned vectors — renderers must finish using it before `submitFrame` returns or copy what they need. Today `submitFrame` is called synchronously from the sim thread immediately after the swap; the atomic swap is the synchronization point if a future renderer reads from another thread.

`RenderFrame::alpha` is the wall-clock fraction (0..1) past the last committed tick. `step()` always submits with `alpha=0`. `run()` additionally calls `submitInterpolatedFrame(alpha)` which mutates `alpha` on the current front frame and re-submits — no rebuild, since world state is unchanged between ticks. The same method is exposed on the public `Engine` API for hosts that drive `step()` manually and need a fresh submit while paused — `step()` is a no-op when paused (no `submitFrame` call), so renderer-side texture edits (UI overlay bitmap, HUD updates) between paused ticks only become visible if the host calls `submitInterpolatedFrame(0.0f)` itself. See `doc/renderer_integration.md` § "Paused-host re-submit" and `tests/submit_interpolated_frame_test.cpp` for the contract pin.

`RenderFrame::prevTransforms` is a span parallel to `instances`, carrying each render-eligible entity's previous-tick transform (or current on first visible frame). Gives a clean `lerp(prev, current, alpha)` with no spawn special-case. Backed by `prevTransformByIndex_` + `prevTransformGenByIndex_` (flat vectors keyed by entity index; generation guard filters stale entries).

## Render contract (renderer-neutral PODs)

`include/threadmaxx/render/` — every type is a renderer-neutral POD; the engine never positions Camera/Light/etc. as per-entity dense storage. Camera/Light/MeshSkinned/AnimationPose are deliberately NOT per-entity components (cameras/lights are few, AnimationPose is ringbuffered, `RenderFrameBuilder` fits them better than `EntityStorage`; rationale in `doc/render_contract.md`).

- `Camera.hpp` — POD with `ProjectionMode`, view + projection matrices (column-major). **Projection MUST be Vulkan-style (NDC z ∈ [0, 1]): `m22 = f/(n-f)`, `m32 = fn/(n-f)`.** GL-style (`m22 = (f+n)/(n-f)`) clips near-camera vertices in Vulkan and produces visible artifacts on close geometry. `RenderFrame::cameraIndexById(id) -> optional<uint8_t>` maps back to the index in `RenderFrame::cameras` (= bit in `cameraMask`); cap `kMaxCameras = 32`.
- `Light.hpp` — `LightType` (Directional/Point/Spot), color/intensity, range, cones, `castsShadow`.
- `DrawItem.hpp` — `DrawItem` + `MeshSkinnedRef` / `AnimationPoseRef` / `MaterialOverride`. `cameraMask` is a 32-bit bitset.
- `DebugGeometry.hpp` — `DebugLine`, `DebugPoint`, `DebugText`. `DebugText::text` is a borrowed `string_view`; producer owns lifetime to the next frame swap. `RenderFrameBuilder::addDebugText(pos, string_view, color)` has an owning-string overload that copies into a per-builder arena. **M7.2 — `DebugLine` and `DebugPoint` carry a `std::uint32_t cameraMask = 0xFFFFFFFFu`; bit `k` ↔ `RenderFrame::cameras[k]`.** Default all-ones = visible from every camera (backwards compatible). Renderers that support per-camera filtering MUST honor the mask; renderers that don't MUST treat any non-zero value as "visible" so the renderer-neutral default still works. 32-bit width matches `kMaxCameras = 32`. The reference Vulkan renderer groups vertices per-camera in `recordFrame` and draws only the matching slice in `recordCamera`; pre-M7.2 callers see no behavior change since the default mask covers every camera.
- `RenderPasses.hpp` — `RenderPass { Opaque, Transparent, ShadowCasters, Overlay }`, `kRenderPassCount = 4`, `passIndex(p)`.
- `RenderFrameBuilder.hpp` — per-system slice. Move-only; engine owns one per system, reset before each tick, merged after every postStep + event drain.
- `Visibility.hpp` — `extractFrustum(camera) -> Frustum`, `intersectsFrustum` AABB p-vertex test, `cullByFrustum(items, bounds, cameras)` writes `cameraMask` in place (33rd camera silently dropped).
- `InstanceBufferLayout.hpp` — `alignas(16) InstanceLayoutEntry` (128 B, std140-friendly). `packInstance(item)` / `packInstances(items, dst)`.
- `UploadRing.hpp` — header-only frame-to-frame bump allocator. `setGrowOnOverflow(true)` allows oversize allocations.

## ISystem::buildRenderFrame hook

`ISystem::buildRenderFrame(RenderFrameBuilder&)` is the render-prep hook. Default: empty. Invoked after every `postStep` and event drain, on the sim thread, single-threaded, in registration order. Each registered system gets its own builder (engine-owned, persisted across ticks for zero-alloc steady state); the engine merges every builder into the back frame.

`RenderFrame` exposes both paths simultaneously: the flat `instances` lane (auto-populated from `RenderTag + !DisabledTag` entities) AND the hierarchical fields (populated via the hook). The two coexist with no overlap. Headless renderers consume just `instances`; richer renderers consume the hierarchical fields.

`SystemStats::buildRenderFrameSeconds` is the per-system timing for the hook (reset to 0 each tick, including across paused steps).

## Bundles, spawning, World query sugar

- `bundle(Cs...)` (`CommandBuffer.hpp`) is a variadic factory returning a `Bundle` with the supplied values and `initialMask` = union of the bits. Feed to `CommandBuffer::spawnBundle(...)`.
- `Bundle::with<T>(value)` is the builder method that sets the field AND the matching presence bit in one call. Composable: `Bundle{}.with(Transform{...}).with(Health{...})`.
- Default-mask `spawn(...)` auto-derives `RenderTag` (from `meshId >= 0`) and `Parent` (from `parent.valid()`). The §3.1 batch-5 setters (`setHealth` / `setFaction` / `setAnimationStateRef` / `setPhysicsBodyRef` / `setNavAgentRef` / `setBoundingVolume`) attach their presence bit on commit; detach via `cb.removeTag(e, Component::Foo)` or `cb.setComponentMask`. Default `spawn` does NOT attach these bits — opt in via `Bundle` or the explicit-mask overload.
- `world.has<T>(handle)` / `world.get<T>(handle)` — header-only sugar for the built-in types. `get` asserts presence. Tag-only categories (`StaticTag`, `DisabledTag`, `DestroyedTag`) reject `has<T>` at compile time — use `world.hasTag(handle, Component::Tag)` instead.

## ComponentSet width & tag-only components

`ComponentSet` is `std::uint64_t`; 16 bits currently allocated (Transform=0 … DestroyedTag=15). 48 spare bits before another widening. Serialization writes the full 64-bit field; v1 32-bit snapshots are not loadable.

`StaticTag` / `DisabledTag` / `DestroyedTag` have no dense storage and no commands. `cb.addTag` / `cb.removeTag` race-free single-bit flips that compose across workers. `buildRenderFrame` skips `DisabledTag`; the engine never auto-destroys on `DestroyedTag` — it's a gameplay-only marker. Tag bits round-trip through `WorldSnapshot`.

## Engine subsystems

### ResourceRegistry

`include/threadmaxx/Resource.hpp` — engine-owned, thread-safe under a single internal mutex. Values are type-erased `std::shared_ptr<void>` keyed by `std::type_index`. Sized for setup-time registration and per-frame lookups, not high-throughput concurrent inserts — async loaders should do I/O off-thread and call `add()` when ready. Stale handles never alias new ones (slot generation bumps on every `remove`).

Opt-in refcount path: `addRefCounted<T>(value)` returns a `ResourceHandle<T>` (refcount=1); `acquire(id)` bumps; `~ResourceHandle` decrements; on zero refcount the slot is freed AND its generation bumps. Legacy-added slots are marked `refCounted=false` and `acquire` refuses them. `refCount(id)` reports 0 for non-refcounted or stale slots. `ResourceHandle<T>::get()` / `operator->` / `operator*` are the indirection sugar (`get` returns `const T*` or nullptr).

### Resource loaders + hot reload + preload

`IResourceLoader` (`Resource.hpp`) is the per-tick pump for asset I/O. Registered via `Engine::addResourceLoader`. The engine calls `loader->update(engine)` once per `step()` after the last `postStep` commits and before the reservation reap / event drain. Teardown in reverse-registration order during `shutdown()`. Engine never spawns threads for loaders — each owns its own pool.

Optional virtuals: `onShutdown(Engine&)` (reverse-registration teardown), `markStale(index, generation, type)` (dispatched by `Engine::markResourceStale<T>(id)`; loader publishes `AssetReloaded{old/new index+gen, type}` on `events<AssetReloaded>()` after installing the replacement — use `AssetReloaded::matches<T>(id)` to filter), `cancel(Engine&)` (called before `update()` so a loader can drop newly-stale requests in the same tick; `LoaderStats::cancelled` aggregates), `stats() -> LoaderStats` (pending/inFlight/ready/failed + memory; `Engine::aggregateLoaderStats()` sums them).

`Engine::preloadUntil(predicate, timeout = 5s)` pumps every loader's `update()` in a yield loop until predicate or timeout. Simulation doesn't advance. Sim-thread only.

### HierarchySystem

`makeHierarchySystem(HierarchyConfig{})` propagates world transforms for `Parent`-attached entities. Runs single-threaded in `ctx.single()`; DFS-with-memoization plus a cycle-safety `onStack` guard. Scratch state is system-member, reused across ticks. Composition: `position = parent.position + rotate(parent.orientation, local.position)`; `orientation = parent.orientation * local.orientation`; `scale = local.scale` by default (`HierarchyConfig::propagateScale = true` opts in to component-wise chaining). See `doc/hierarchy.md`.

Register *after* any user system that writes `Transform`. Both declare `writes = {Transform}` → different waves; hierarchy sees the writer's commits.

### JobSystem

`src/JobSystem.{hpp,cpp}` — per-worker work-stealing deques. Submit round-robin via an atomic counter; idle workers `try_lock`-steal LIFO from siblings' back. `waitIdle` synchronizes on a separate mutex/CV. `outstanding_` is `acq_rel` for cross-thread visibility; `notify_all` on `doneCv` when it hits zero.

**Wake contract (2026-05-25, missed-wakeup fix).** Workers park on per-worker CVs (one futex per worker keeps contention low at high pool sizes). The pre-2026-05-25 design had a missed-wakeup hole: when a worker was blocked in a user wait (e.g. `JobLatch::wait` inside a nested `parallelFor`), `submit()`'s `notify_one(target.cv)` was wasted, and other parked workers stayed parked because their predicate (`self.hasWork() || stopping_`) returned false on any wake whose work landed on a sibling queue. They re-parked without ever calling `trySteal`. Result: stranded work, intermittent ~1/5 hang on rpg_demo `--stress` at 71 workers. Three pieces close the hole without re-introducing contention:

1. **Monotonic `wakeSeq_` atomic** — `submit()` bumps it (release) AFTER the queue push. Workers snapshot `wakeSeq_` BEFORE incrementing `parkedCount_` and entering `cv.wait`. The predicate is `self.hasWork() || stopping_ || wakeSeq_ != snapshot`. Any submit anywhere advances `wakeSeq_`, so a notified worker exits `cv.wait` and goes back to `popOwn` + `trySteal` at the top of `workerLoop`. If their own queue is empty and steal finds nothing, they re-snapshot and re-park.
2. **`parkedCount_` atomic** — incremented around the park bracket. `submit()` uses it to gate the helper notify (next bullet); skipped when zero (all workers busy → no fan-out cost in the steady-state hot path).
3. **Helper notify at `(target + 1) % n`** — when `parkedCount_ > 0`, `submit()` does one extra `notify_one` on the helper's CV in addition to `target.cv`. Over a wave of N submits the helper rotates through every worker, so any reachable parked worker is notified at least once even if their own queue gets no direct push. Combined with the `wakeSeq_` snapshot, this also closes the race where a worker is mid-park transition when a submit arrives (the submit's `wakeSeq_++` happens-before the worker's predicate read; the predicate observes the advance and skips `cv.wait`).

Pinned by `tests/job_system_missed_wakeup_test.cpp` (200 nested-parallelism iterations at every worker count from 4 to `hardware_concurrency`). Empirical: rpg_demo `--stress 300` at auto (71) workers — pre-patch with bug 31.7 s / sys 3.1 s; this fix 29-30 s / sys 2.9-3.0 s; intermediate "semaphore + fan-out" forms regressed to sys 49-174 s due to either single-futex contention or N-wide notify storms. **Don't reintroduce a single global wake futex** (semaphore or otherwise) — at 71 workers the contention costs ~50× the per-worker-CV design's sys time. **Don't drop the `wakeSeq_` leg of the predicate** — workers will re-park on any non-direct-queue wake and strand sibling work.

Per-worker `ownPops` / `stolenJobs` / `histogram` are `std::atomic<uint64_t>` with relaxed ordering — when adding a new per-worker counter follow that pattern. Per-worker counters are touched by the owning thread (own pop/histogram) or by the stealing thread (writing its OWN slot for `stolenJobs`); reads from `stats()` happen from any thread.

`JobSystemStats::jobDurationHistogram` is 16 log2-µs bins; `JobSystem::binFor(ns)` is static for testability. `JobSystem::outstanding()` reports live in-flight for queue-depth sampling.

Determinism is preserved because commits are submission-ordered (worker execution order doesn't matter). Stress test: `tests/job_system_stress_test.cpp`.

**`JobLatch` is the TSAN-clean replacement for `std::latch`.** Lives in the anonymous namespace at the top of `src/EngineImpl.cpp`. libstdc++'s `std::latch::wait()` lowers to a futex that ThreadSanitizer cannot see through. `JobLatch` uses `std::ptrdiff_t` under a mutex+CV — every `count_down` takes the lock. **Don't re-introduce `std::latch` here without a TSAN re-run; don't re-introduce the lock-only-on-last-decrement form without 100× stress validation under TSAN** (the "optimized" variant had a missed-wakeup hazard).

`JobPriority { High, Normal, Low }` (`System.hpp`). `parallelFor` overloads accept it; no-priority overloads default to `Normal`. Workers scan deques in priority order. Backwards-compatible — collapses to single-deque behavior when every job is `Normal`.

### Lifecycle hooks

`ISystem` exposes three: `preStep` (serial, registration order, before any wave), `update` (parallel within a wave), `postStep` (serial, registration order, after every wave). `preStep`/`postStep` default to no-op; commits flush immediately on the sim thread.

### ScratchArena

`include/threadmaxx/ScratchArena.hpp` is a chained-slab bump allocator. Pass via the three-arg `parallelFor` / `single` overloads alongside the `CommandBuffer`. Slabs survive across waves; reset rewinds the bump pointer. `allocate<T>` static_asserts that `T` is trivially destructible. Wave-scoped lifetime, not engine-scoped.

### EventChannel

`include/threadmaxx/EventChannel.hpp` — typed double-buffered queue. `Engine::events<T>()` is the factory; channels are owned in `EngineImpl::eventChannels_` keyed by `std::type_index`. The map is mutex-guarded. **Warm cross-thread channels at setup** by calling `events<T>()` on the sim thread before workers / watchdogs touch them.

`emit` is lock-free (atomic Treiber stack with CAS prepend; `exchange(nullptr)` + reverse on drain restores per-thread FIFO). Subscriber list is mutex-guarded but only touched on subscribe / unsubscribe. `pendingCount()` is approximate under concurrent emit.

Subscriptions: `subscribe(fn) -> SubscriptionId` / `unsubscribe(id)` for persistent; `subscribeScoped(fn) -> Subscription` for RAII (move-only, type-erased, auto-detaches; holds a `weak_ptr` so it safely no-ops if the channel dies first; `reset()` / `valid()`). Tick-end `drain()` invokes each callback once per back-buffered event BEFORE the front/back swap. Subscriber list is snapshotted before invocation — a callback that subscribes/unsubscribes doesn't invalidate iteration.

### Pause & time scale

`Engine::setTimeScale(s)` clamps negative `s` to zero; `dt` becomes `fixedStepSeconds * s`. `tick()` and `simulationTime()` advance independently of time scale. `setPaused(true)` makes `step()` a no-op (zeros per-tick stats); `run()` keeps re-submitting the current front frame so the renderer doesn't freeze.

### Reserved handles

`Engine::reserveEntityHandle()` / `SystemContext::reserveHandle()` allocate via `EntityStorage::reserveHandle()` under `reservationMtx_`. `EntityHandle::valid()` is true immediately but `world.alive(handle)` is false until `materializeReserved` runs during commit (triggered by `cb.spawn(handle, ...)`). Unmatched reservations are reaped in `discardAllReservations` at step end (bumps generation, returns slot to free list).

Batch form: `Engine::reserveEntityHandles(count, span)` / `SystemContext::reserveHandles(count, span)` / `EntityStorage::reserveHandles` acquire the mutex once. Returns the number actually written (`min(count, span.size())`).

### SpatialHash

`SpatialHash<Payload>` (header-only, `SpatialHash.hpp`) — uniform-grid index keyed by 3D cell coordinate. Not thread-safe: insert from a single thread (usually `preStep`); queries safe from workers once built. `clear()` keeps bucket allocation so steady-state per-tick rebuilds pay zero allocations after the first tick. Engine doesn't own one — user composes it.

### Serialization

`include/threadmaxx/Serialization.hpp` is header-only. Per-component `serialize` / `deserialize` for every built-in plus `Vec3`, `Quat`, `ComponentSet`, and the `WorldSnapshot` POD. `World::snapshot()` (`World.cpp`) copies the engine's dense arrays into a `WorldSnapshot`. Binary format: `[magic 'SXMT'][version u32][count u64][N dense arrays each prefixed by u64 length]`; deserialize rejects mismatched magic/version. `kWorldSnapshotVersion = 2`. Restoration is game-side via `cb.spawn(...)` — the engine never bypasses commit.

### Logger

`ILogger` (`Logger.hpp`) is a single `log(level, message)` virtual. The engine routes startup/shutdown/registration/loader-error messages through whichever sink is installed via `Engine::setLogger`; `nullptr` restores `DefaultLogger` (writes to `std::cerr` at `Warn+`). `EngineImpl::logger()` returns the active sink. The engine doesn't take ownership.

### registerSystemAt

`Engine::registerSystemAt(position, system)` inserts at a specific index in registration order (clamped to `[0, registeredSystemCount()]`). Calls `onRegister(world)` and `rebuildWaves()`. Returns the actual insertion index. For tests / mod loaders that need to inject a system at a known point.

### IRenderer::onResize

`IRenderer::onResize(w, h)` is a default-no-op virtual; `Engine::notifyResize(w, h)` forwards window resize events into the currently installed renderer. No engine state changes. Sim-thread by convention.

`Engine::workerCount()` is the public accessor for the worker pool size — cheaper than going through `jobSystemStats().workerCount` (no atomic loads / no histogram merge). Use for sizing per-worker scratch and ring-buffer slots.

## Task graph (named tag edges) & wave scheduler

Three optional virtuals on `ISystem`:
- `dependencies() -> span<const TaskTag>` — tags this system *consumes*.
- `provides() -> span<const TaskTag>` — tags this system *produces*.
- `preferredGrain() -> uint32_t` — hint to `pickGrain` when `parallelFor` passes `grain=0`. Pinned per-system.

`TaskTag { string_view name; uint64_t hash; }` (`TaskTag.hpp`) — constexpr FNV-1a hash; `operator==` compares `name`. Hash collisions degrade to extra string comparisons, never to wrong scheduling.

`EngineImpl::rebuildWaves` does topological-sort-then-pack:
1. Edges: rw-conflict `a < b` → `a → b`; tag overlap `a.provides ∩ b.dependencies` → `a → b` (both directions checked).
2. Kahn picks zero-in-degree systems in registration order. On stall (tag cycle), the lowest-index stuck system has its tag-only incoming edges dropped, logged via `ILogger`, Kahn resumes.
3. Greedy wave-pack: earliest wave where predecessors are done AND no rw-conflict with anyone in the wave.

Bit-for-bit unchanged when no tags are declared. With tags, consumers may land later than first-fit would have given; producers are never delayed by consumers.

`Engine::taskGraphSnapshot() -> std::vector<TaskGraphNode>` exposes the graph for HUD diagnostics / Graphviz / test assertions.

## Adaptive tuning (ADAPTIVE_TUNING.md)

`include/threadmaxx/Tuning.hpp` is the public surface for the Phase-T tuner. Five types: `SystemGrainOverride { string systemName; uint32_t preferredGrain; }`, `TuningPatch { vector<SystemGrainOverride> grainOverrides; }`, `ITuningPolicy { void observe(EngineStats, span<SystemStats>, JobSystemStats); optional<TuningPatch> propose(); }`, `enum class TuningMode { Off, Active, Scripted }`, and `TuningTrace` (record/tryGet/serialize/deserialize). `Engine::setTuningPolicy(p)` / `tuningPolicy()` install / inspect; `setTuningPolicy(nullptr)` detaches and discards any staged patch.

**Call sites in `step()`** — `observe()` and `propose()` fire once per step AFTER the trace-sink callback (same `EngineStats` / `SystemStats` / `JobSystemStats` view a sink would see). Any returned non-empty patch is stored in `EngineImpl::pendingPatch_`. `applyPendingTuningPatch()` runs at the top of the NEXT `step()` BEFORE preStep — never mid-wave. Patch application overwrites `EngineImpl::systemPreferredGrain_[i]` for each matched name (linear name scan); unknown names log `[tuning] grain override for unknown system 'X' ignored` at Warn. Applied changes log `[tuning] 'X' preferredGrain old -> new` at Info.

**Determinism contract** — same input + same scripted patch stream = bit-identical `commitHash` stream (tuning is a scheduling knob; never touches storage or commit order). The mechanism cost when no policy is installed is one null-pointer check per step.

**`TuningMode` + `TuningTrace`** (T6) — `Engine::setTuningMode(mode)` / `tuningMode()` selects the runtime mode; `Engine::setTuningTrace(t)` / `tuningTrace()` attaches a non-owning trace. `setTuningPolicy(p)` implicitly switches to `Active` when `p != nullptr` and `Off` otherwise — pre-T6 callers stay backwards-compat. In `Active`, every applied patch is appended to the attached trace keyed by `EngineStats::tick` at the moment of the call. In `Scripted`, `policy.propose()` is ignored entirely; staged patches come from `trace.tryGet(currentTick)`. `Off` is a single null-check on the hot path. Wire format is `[magic 'TUNE' u32][version u32][entryCount u64]` followed by per-entry `[tick u64][overrideCount u64]{[nameLen u64][name][grain u32]}*` — host-endian POD, same caveats as `WorldSnapshot`. Bench gate: `bench/adaptive_tuning_bench --mode=scripted` lands identical commitHash for all measured ticks (last empirical: −7.44% scripted-vs-Active timing, well inside the +2% gate; the Active path pays observe/propose overhead Scripted skips).

**`AdaptiveGrainPolicy`** (`include/threadmaxx/AdaptiveGrainPolicy.hpp`, header-only). Built-in `ITuningPolicy` that adjusts per-system `preferredGrain` toward a hold band `[targetSubJobMicros/2, targetSubJobMicros*4]`. Per-system EWMA of `SystemStats::avgSubJobMicros` drives the direction; hysteresis = `minSamplesPerChange` same-direction observations before a fire + `cooldownTicks` between consecutive fires on the same system. ε-greedy random one-step walks (probability `explorationEpsilon`) escape stuck local minima. RNG is seeded deterministically (`randomSeed ^ currentTick`) so two runs against identical input produce identical patch streams. Config defaults (`cooldownTicks=60`, `targetSubJobMicros=200`, `stepSize=1.5`, `initialGrain=64`) are tuned conservatively — slow to act, no oscillation under noisy workloads. Bench gate: `bench/adaptive_tuning_bench --mode=tiny-fanout` lands within +10% of the offline grain-sweep optimum (last empirical: +0.14%).

## Cancellation, budgets, priorities

**Tick budget + skip policy.** `Engine::setTickBudget(seconds)` caps the wall-clock per-tick spend. The engine samples after every wave; if elapsed > budget AND `setSkipPolicy(SkipPolicy::Budget)` (the default), `overBudget_` flips and subsequent waves' `ISystem::skippable()` systems are skipped. `preStep` / `postStep` / `buildRenderFrame` are NEVER skipped.

`SkipPolicy::Scripted` ignores `tickBudget_` and consults `scriptedSkips_` (populated via `Engine::pushScriptedSkip(tick, name)`). Networked / lockstep games run `Budget` on the authoritative server, drain `EventChannel<SystemSkipped>`, broadcast the log, clients replay with `Scripted` — world hashes match.

`SystemContext::shouldYield()` exposes the over-budget flag — a single atomic load, cheap to poll inside long `parallelFor` bodies.

`SystemSkipped { tick, systemName, reason }` event (`SkipPolicy.hpp`) emitted on skip; reason is `"budget"` or `"scripted"`.

## Commit path + hash determinism (v1.3)

`Config::singleThreadedCommit = true` (default) — wave commits go through `commitBuffer`. Pre/postStep hooks always do regardless.

`Config::singleThreadedCommit = false` opts into `commitBuffersSharded` (three passes: build migrating-entity set; sim-thread classify + apply migrations + bin value-only commands per chunk; parallel-apply bins). **Sharded never beats single-threaded on tested workloads** (serial classifier ~130 ns/cmd exceeds 4-way apply's 50 ns/cmd recovery) — keep the default. Documented fallback under profiler-confirmed contention. Auto-falls-through to single-threaded when `totalCommands < 256` OR `totalValueOnly == 0` OR `archetypeChunkCount() < 2`.

**Per-archetype hash rollup is the v1.3 commit-hash contract** (`Config::legacyCommitHash = false`, the default). Every `ArchetypeChunk` carries `mutable uint64_t cachedHash + mutable std::atomic<bool> hashDirty`; `EngineImpl::finalizeCommitHash()` re-hashes dirty chunks at end of step (parallel when ≥ 2 dirty), folds sorted-by-`mask.bits()` into `EngineStats::commitHash`. v1.2 → v1.3 contract: "same command stream" → "same final per-archetype state." External clients must re-record reference hashes; `Config::legacyCommitHash = true` keeps the v1.x byte-mix during deprecation (slated for removal in v1.4).

**Chunk dirty-marking is load-bearing.** Every mutation path MUST set `chunk.hashDirty.store(true, std::memory_order_relaxed)`. Covered: `ArchetypeTable::insert` / `removeSwapPop` / `migrate` (the latter via the other two), every `EntityStorage::mut*()` setter. When adding a new chunk-mutating method to `ArchetypeTable`, set dirty at the end. `hashDirty` is `std::atomic<bool>` with relaxed ordering — required because the S10 row-split path can put multiple workers on disjoint rows of the same chunk, all writing `true` concurrently. The write is idempotent so relaxed is sufficient; the read in `finalizeCommitHash` is synchronized via the `JobLatch` mutex acquire that ends Pass C. `ArchetypeChunk` defines explicit copy/move ctors that load+store the atomic so it remains usable in `std::vector<ArchetypeChunk>`.

`hashChunkContent` is FNV-1a-64 over: `mask.bits` → `entities.size` → `entities[0..]` → each built-in dense vec in component-enum order (mask-gated) → each user column in ascending-bit order (`col.bit`, `col.stride`, `col.bytes`). `finalizeCommitHash()` sorts chunks by `mask.bits()` ascending before folding (empty chunks included).

Adding a new command variant requires arms in both `applyCommandImpl` and `hashCommandImpl` at the top of `src/EngineImpl.cpp`. Both commit paths go through these helpers — a missing arm fails to compile rather than silently corrupting state. New variants auto-fall-through to the global lane (`commandIsMigrating` returns true for anything not in `CmdSetTransform / CmdSetVelocity / CmdSetAcceleration / CmdSetUserData`).

`Config::logCommitHashEvery = N > 0` logs `commitHash tick=<T> hash=0x<16hex>` via `ILogger@Info` every N ticks — pinpoints a divergent tick for local repro. Default `0` is silent / zero-cost.

**Migration batching (SHARDED_OPTIMISATION.md S6).** When a contiguous run of `CmdAddTag` / `CmdRemoveTag` / `CmdSetHealth` / `CmdSetFaction` / `CmdSetBoundingVolume` commands all target the same (srcArch, dstMask) pair AND the run is ≥ `Config::batchMigrateThreshold` (default 16), the run is dispatched through `EntityStorage::setMaskAndMigrateBatch` instead of N individual `setMaskAndMigrate` calls. Underneath, `ArchetypeTable::migrateBatch` inserts in submission order then pops in descending srcRow order — the descending-pop order is the determinism-preserving invariant proven in `tests/migration_batch_test.cpp`. Fires from both `commitBuffer` (serial) and `commitBuffersSharded` Pass B's global lane. `CommitBreakdown::batchedMigrations` exposes the per-step count for benches/tests. Set `batchMigrateThreshold = std::numeric_limits<std::uint32_t>::max()` to disable (used by bench A/B); below ~4 the per-cmd path wins.

**Record-time per-chunk routing (SHARDED_OPTIMISATION.md S8).** When `cfg.recordTimeRouting == true` (default) AND sharded commit is on, the engine installs a chunk-locator hook on every fresh `CommandBuffer` it hands to a worker. Each value-only recording method (`setTransform` / `setVelocity` / `setAcceleration` / `setUserData`) calls the locator, gets the entity's current archetype index, and pushes the command's index into `chunkBuckets_[arch]`. Migrating cmds and stale handles go to `globalIdx_`. The sharded commit's Pass A consumes only `globalIdx_` (a small fraction of total cmds for typical workloads); Pass B walks buckets directly and demotes any entry whose target is in the cumulative wave-migrating bitmap. The bitmap is **wave-cumulative** (cleared at wave start in `step()`, not per-commit) so cross-system migrations within a wave correctly demote stale-binned cmds in later systems' commits. Set `cfg.recordTimeRouting = false` (or `THREADMAXX_NO_ROUTING=1` env var in benches) to fall back to the pre-S8 full-cmd scan. Ignored when `singleThreadedCommit = true` — serial path pays zero S8 overhead.

**Sim-thread-inline largest bin (SHARDED_OPTIMISATION.md S9).** When `cfg.inlineLargestBin == true` (default) AND sharded commit is on, Pass C identifies the single largest large-bin and runs it inline on the sim thread; only `largeBins − 1` worker jobs go to `JobLatch`. The sim is now a peer lane rather than a wait barrier — `wait_us` drops to near-zero on every routing-active workload. Commit wall clock improves on single-large-bin workloads (sim avoids the latch + worker cache miss path, ~−25% commit_us on `setTransform/Churn` etc.) but does NOT improve on balanced multi-large-bin workloads (e.g. `MultiArch` with 4 ≈25k-cmd bins on 4 workers: workers still take time T, sim's 4th lane also takes T, wall clock = T). Determinism unchanged — bins target disjoint chunks. Set `cfg.inlineLargestBin = false` (or `THREADMAXX_NO_INLINE_LARGEST=1` env in benches) to revert. `CommitBreakdown::inlineLargestApplied` counts qualifying calls.

**Row-split largest bin (SHARDED_OPTIMISATION.md S10, OPT-IN, default OFF).** `Config::splitLargestBin` opts Pass C into partitioning the single largest bin into `min(workerCount + 1, largestBinSize / kMinBinForJob)` row-range sub-bins; sub-bin 0 runs inline on sim, the rest on workers. Eligibility additionally requires `largeBins == 1`. The implementation is determinism-preserving (each cmd routes to the sub-bin whose row range covers its target row; `pass_c_split_test.cpp` pins the contract) but the default is **off** because the bench shows the partitioner's `std::visit` + `locate()` cost (~25–30 ns/cmd) exceeds the apply cost (~13.6 ns/cmd) — the only S10-eligible bench workload (`setTransform/SingleArch`) regresses +207%. Preserved as a fixed point for a future revisit with record-time row-bucketing (the cheap-classifier prerequisite for an S10 that wins). See `SHARDED_OPTIMISATION.md` § "S10 outcome" for the diagnosis.

**JobLatch spin-before-block (SHARDED_OPTIMISATION.md S11).** `Config::jobLatchSpinIters` (default `4096` ≈ 10-40 µs) gates a hot-loop in `JobLatch::wait()` over an atomic `done_` flag (release-stored by the final `count_down` decrementer) before falling back to the mutex+CV blocking path. When the spin observes `done_` within budget, the kernel-sleep / wakeup-IPI of `cv_.wait` is skipped — but the mutex is ALWAYS re-acquired before returning so a stack-allocated latch's destructor cannot race the worker's still-in-flight `cv_.notify_all` / `lock_guard` dtor. The mutex acquire is uncontended on the win path (~30 ns). Empirical: `setTransform/MultiArch` commit_us −22% (Pass C wait 641 µs → 125 µs), `setTransform/Churn` commit_us −9%, no detectable regression on no-wait workloads. Determinism unchanged. Set `cfg.jobLatchSpinIters = 0` (or `THREADMAXX_NO_LATCH_SPIN=1` env in benches) to revert.

**Workload-aware auto fallthrough (SHARDED_OPTIMISATION.md S16, OPT-IN, default OFF).** `Config::workloadAwareCommit` (default `false`) + `Config::workloadAwareGlobalPercent` (default `30`) add a fourth pre-condition to `commitBuffersSharded`'s early-out: when on AND `globalCount * 100 >= totalCommands * workloadAwareGlobalPercent`, the call falls through to per-buffer serial commit. Zero record-time cost (uses counters already maintained for free). `CommitBreakdown::workloadAwareFallthrough` is the per-step fire counter. `tests/workload_aware_commit_test.cpp` pins both directions plus a determinism check across the gate. Default 30% is conservative — only trips on heavily-mixed workloads (≥30% global); RPG-mix's actual global ratio is ~4%, so the default does NOT trip on it. The mechanism is correct (PCT=1 demonstrates the gate firing on RPG-mix, commit_us drops to the serial baseline), but the count-based signal can't distinguish a workload that's time-dominated by few-but-expensive globals from a balanced one — users tuning for RPG-mix-shape signal must lower `workloadAwareGlobalPercent` to ≈3. Env-var overrides: `THREADMAXX_WORKLOAD_AWARE=1` + optional `THREADMAXX_WORKLOAD_AWARE_PCT=N`.

## Stitched-cache safety

`EntityStorage::ensureStitched()` is mutex-protected (double-checked locking on `stitchedDirty_`). Fast path (cache clean) only pays the atomic acquire on `stitchedDirty_`; slow path (rebuild) is serialized. Required to make `world.transforms()` etc. safe from many concurrent worker reads in the same wave.

## WorldView

`SystemContext::worldView() -> const WorldView&` — wave-scoped read-only view of chunk storage. Caches chunk pointers + entity counts at wave start; shared across every system in the wave. Engine rebuilds it between waves (commits may shift chunk inventory). Capture by reference and reuse across multiple `parallelFor` / `single` calls without paying repeated indirection through `World::archetypeChunk(i)`. `EngineImpl` rebuilds `worldView_` inside `step()` before preStep, between preStep commits, at the top of every wave, and between postStep commits.

## MaskCache

`Query.hpp::MaskCache` — user-owned vector of dense indices matching a `ComponentSet`. `forEachWithCached<...>` iterates the cache, skips out-of-range entries, no per-entity mask test in the hot loop. Rebuild discipline is on the caller: `cache.rebuild(world, required<T...>())` in `preStep` when the required set may have a new or removed entity. The bounds check protects against destroys between rebuilds but cannot detect a mask flip to no-longer-match. `MaskCache::reserve(n)` / `capacity()` pre-warm the index storage; `clear()` preserves capacity.

## Sub-job dispatch in forEachChunk

`forEachChunk<Required...>` splits oversized chunks into multiple sub-jobs so workers fan out across the rows of a single huge chunk. `kForEachChunkSubJobThreshold = 1024` (public constexpr in `Query.hpp`); chunks at or below stay a single job, above splitting kicks in. `detail::chunkSubJobBudget(rowCount, workers)` picks per-sub-job rows as `max(threshold, ceil(rowCount / (workers * 8)))`. The callback fires once per sub-job; spans across sub-jobs sum to the full chunk's rows; every row visited exactly once. `SystemContext::workerCount() -> uint32_t` is the cached pass-through to `Engine::workerCount()` used to compute the budget.

## Async snapshot / trace sink

`FileTraceSink::setAsync(bool)` spawns an internal writer thread + queue + mutex/CV. `onFrame` copies the borrowed `FrameSnapshot` into owned storage (`OwnedFrameSnapshot` in `Telemetry.cpp` is the bridge), enqueues, returns. Joined on `setAsync(false)`, `onShutdown`, or destruction.

`Engine::snapshotAsync(callback)` captures `world().snapshot()` synchronously on the sim thread (the snapshot copy must be sync — `WorldView` caches chunk pointers, not contents, so a mid-snapshot commit would race), then posts the user's callback onto a dedicated engine-owned background thread (`snapshotWorker_`, lazily spawned, joined in `shutdown` / `~EngineImpl`).

Snapshot consistency: reflects state at the moment `snapshotAsync` was called. Callbacks must not call back into the engine's mutation API. When adding new async I/O sinks, follow the same pattern: deep-copy the borrowed span, enqueue, drain on a dedicated worker, join in dtor.

## Telemetry / FrameSnapshot

`Engine::frameSnapshot()` returns `FrameSnapshot{EngineStats, span<const SystemStats>, JobSystemStats}` — consistent across the three. `writeJsonLines(os, snap)` serializes one snapshot as a single `\n`-terminated JSON object with `"commit_hash":"0x…"` (16-char hex). `ChromeTraceWriter` is the streaming companion: ctor writes `[`, `emit(snap)` appends one `step` record + one per system per call (`args.commit_hash` carries the hash), dtor writes `]`. Systems span is invalidated by `registerSystem` / `registerSystemAt` / `shutdown`.

`SystemStats` extras: `waitSeconds` (latch wait time across `parallelFor`), `peakQueueDepth` (max `JobSystem::outstanding()` right after submit), `buildRenderFrameSeconds`, plus the ADAPTIVE_TUNING.md T3 pair `avgSubJobMicros` (1/16-EWMA of per-sub-job lambda µs; persists across steps, 0 if the system never called `parallelFor`) and `subJobsLastStep` (parallelFor sub-jobs dispatched this step). Both fields are emitted in `writeJsonLines` (`avg_sub_job_us`, `sub_jobs`) and in `ChromeTraceWriter` per-system args, and surfaced in `perf_audit_rpg_demo`'s top-systems table.

`include/threadmaxx/Telemetry.hpp` opt-in extras:
- `ITraceSink::onFrame(FrameSnapshot)` — installed via `Engine::setTraceSink(ITraceSink*)`. Engine never takes ownership; sink must outlive the engine.
- `FileTraceSink` — rolling Chrome-trace JSON. Rotates when file exceeds `Config::rotationBytes` (default 64 MiB); `pathTemplate` may contain `%N` or be a stem. Silent no-op on open failure.
- `HudTraceSink` — seqlock-protected latest-snapshot sink. `tryGet(out)` is lock-free and torn-write-free. **The seqlock has an inherent TSAN-visible race on `data_` BY DESIGN; suppressed in `cmake/tsan.supp`. Don't "fix" it with mutexes** — that defeats the purpose. Activate with `TSAN_OPTIONS=suppressions=<path>/cmake/tsan.supp`.
- `FrameBudgetWatcher` — `ISystem` whose `postStep` emits `BudgetExceeded` when `lastStepSeconds` > target.
- Stall watchdog — `Engine::setStallTimeout(seconds)` spawns a background thread that emits `EngineStall` via `events<EngineStall>()`. `stallTimeoutSeconds_` is `std::atomic<double>` (relaxed). Joined on `setStallTimeout(0.0)` or `~EngineImpl`. Zero overhead when disabled.

Add telemetry from a new subsystem: emit a typed event through `events<MyTelemetryEvent>()` and tell sinks to subscribe.

## Performance patterns

- Prefer flat vectors keyed by entity index over hash maps for hot paths.
- Reuse allocations across ticks via `clear()` / `assign()` to preserve capacity.
- Avoid `std::thread` creation in steady-state code — dispatch through `jobs_->submit` instead (wave-parallel system updates do this).
- Walk chunks via `ctx.worldView()` and `detail::getChunkSpan<C>` rather than the stitched `world.transforms()` view (the stitched view acquires a mutex on first build; chunk walking is mutex-free and 3–5× cheaper at small scales).
- The §3.9.3 batch-18 trick: keep `detail::Command` variant alternatives under 48 bytes when possible. `CmdSpawn` and `CmdAddUserComponent` ride `std::unique_ptr<>` wrappers to keep the variant at 64 B (vs 256 B if both were inlined). The `unwrap` helper at the top of `applyCommandImpl` / `hashCommandImpl` dereferences them transparently. If a new variant would push the variant over 64 B, follow the same pattern.

Empirical (perf_audit_rpg_demo 2026-05-25, normal mode): meanStep 4.43 → 2.47 ms (−44%); static-terrain-render 3.57 → 1.60 ms (−55%). Stress mode: static-terrain-render 0.57 → 0.13 ms (−77%); meanStep run-to-run noisy on the audit box, no clean signal at this scale.

## Sanitizer / soak hygiene

Three sanitizer trees are expected to pass clean: ASAN, UBSAN, TSAN (the last activated with `TSAN_OPTIONS=suppressions=cmake/tsan.supp` for the HudTraceSink seqlock). Sanitizer trees skip benches, the rpg_demo sub-folder, and the SIMD sibling library — "sanitizers run engine-side correctness only" is the standing convention.

The opt-in `tests/concurrency_soak_long.cpp` (50× the ticks of `concurrency_soak_test`, ~5–6 min) is the go-to for surfacing anything that only manifests after sustained steady-state operation.

## When adding a new public symbol

1. Doxygen `@brief`. Load-bearing methods also get `@thread_safety` / `@pre`.
2. Add a `tests/COVERAGE_AUDIT.md` entry if the symbol is non-trivial and not already exercised.
3. If it's an engine knob, add it to `doc/performance_tuning.md` under "the major knobs".
4. If it's a built-in component or a command variant, walk the recipe in `Adding a new built-in component` (especially the §B30 hash-dirty bullet).
5. If it goes into `commitBuffer`'s `std::visit`, add the matching arms in `applyCommandImpl` and `hashCommandImpl`.
